# 1차 수정 제안서 (검증 반영본)

> 현재 상태 안내: 이 문서는 과거 제안/검토 기록이다. 2026-04-28 현재 파이프라인에서는 외부 router, Claude Desktop MCP, watcher/scheduler 자동화, PDF 출력, mem0 원격 저장소를 사용하지 않는다. 현재 기능 범위는 `INDEX.md`, `PIPELINE_OVERVIEW.md`, `코드_로직_문서.md`, `사용법_가이드.md`를 우선한다.

## 목적

현재 코드베이스에서 바로 수정 가능한 범위만 묶어, 실행 경로 일원화와 상태 저장 일관성 확보를 우선 달성한다.

이번 1차 수정의 목표는 아래 3가지다.

1. 동일 입력이 서로 다른 경로로 처리되는 문제를 줄인다.
2. 퀴즈 채점 결과가 한 군데 규칙으로 저장되게 만든다.
3. 리팩터링 없이도 운영 안정성을 높일 수 있는 작은 구조 정리를 진행한다.

## 현재 기준선

이미 반영된 항목:

1. `watcher.py`가 `synthesize.py`만 호출하도록 변경되어 퀴즈 중복 생성 위험을 줄였다.
2. `scheduler.py`가 `generate.py` 대신 `synthesize.py`를 호출하도록 변경했다.
3. `review.py`, `dashboard/quiz_manager.py`가 채점 결과 기록 시 `MemoryManager.record_result()`를 사용하도록 변경했다.

이번 제안서는 이 기준선 위에서 바로 이어서 작업할 수 있는 항목만 정리한다.

## 1차 수정 범위

### A. 구형 weak_concepts 갱신 로직 제거

**배경 및 실제 문제:**

코드 확인 결과 `review.py:81–140`의 `update_weak_concepts()` 함수와 `memory_manager.py:79–146`의 `record_result()` 내부 로직이 **mastery 계산 · priority 매핑 · recent_mistakes 추적을 동일하게 두 벌 구현**하고 있다.

`cmd_report()` (`review.py:288`)는 여전히 `load_weak(pipeline_dir)` 경로로 `weak_concepts.json`을 직접 읽고 있어, `MemoryManager`를 거치지 않는 경로가 남아 있다.

대상 파일:

- `scripts/review.py`
- `scripts/dashboard/quiz_manager.py`

수정 내용:

1. `review.py`에서 `load_weak()`, `save_weak()`, `update_weak_concepts()` 세 함수를 제거한다.
2. `cmd_report()` 내 `load_weak()` 호출을 `MemoryManager.get_weak_concepts()`로 교체한다.
3. `quiz_manager.py`의 주석과 함수 설명을 실제 동작 기준으로 정리한다.

완료 기준:

1. `review.py` 내부에서 `weak_concepts` 직접 계산 코드가 사라진다.
2. `cmd_report()`가 `MemoryManager`를 통해 동일한 데이터를 읽는다.
3. `weak_concepts.json` 쓰기 경로가 `MemoryManager.record_result()` 하나뿐이다.

---

### B. 퀴즈 이동 로직 공용화

**배경 및 실제 문제:**

코드 확인 결과 `review.py:269–282`와 `quiz_manager.py:84–103`의 `complete_quiz()`가 **queue → approved 이동 · vault 사본 삭제 · .md 파일 이동 순서까지 거의 동일한 코드를 각자 보유**하고 있다.

vault quiz 경로 계산도 `review.py:26–36`의 `get_vault_quiz_path()`와 `quiz_manager.py:113–124`의 `_remove_vault_quiz_copy()`에 중복 구현되어 있다.

추가로 `quiz_manager.py:44–54`의 `load_quiz()`가 JSON 파싱 오류를 `None` 반환으로 무음 처리해, 대시보드에서 퀴즈가 조용히 사라지는 버그의 온상이 된다. B 작업 시 함께 수정한다.

대상 파일:

- `scripts/review.py`
- `scripts/dashboard/quiz_manager.py`
- 신규 공용 모듈 1개 (`scripts/quiz_ops.py`)

수정 내용:

1. `review.py`의 `get_vault_quiz_path()` · `remove_vault_quiz_copy()`를 `quiz_ops.py`로 이동한다.
2. queue → approved 이동 · .md 이동 · vault 사본 삭제 로직을 `quiz_ops.approve_quiz()`로 통합한다.
3. `review.py`의 `cmd_do()`와 `quiz_manager.py`의 `complete_quiz()` 모두 `quiz_ops.approve_quiz()`를 호출하도록 교체한다.
4. `quiz_manager.py`의 `load_quiz()`에 예외 처리를 추가해 오류 시 로그를 남기고 `None` 무음 반환을 제거한다.

완료 기준:

1. 승인 처리 로직이 두 파일에 중복으로 남아 있지 않다.
2. 승인 후 `queue` · `approved` · vault `퀴즈/` 상태가 CLI와 대시보드에서 동일하다.
3. `load_quiz()` 실패 시 호출부가 오류 원인을 알 수 있다.

### C. 경로 상수 통합 (점진적)

**배경 및 실제 문제:**

코드 확인 결과 `"퀴즈"` · `"정리"` · `"queue"` · `"approved"` · `"rejected"` · `"cache/papers"` 등의 폴더 이름이 7개 이상의 파일에 리터럴 문자열로 흩어져 있다. `vault / config["notes_dir"] / folder / "정리"` 같은 조합이 최소 3곳에서 독립적으로 계산된다.

**이번 1차에서는 상수 파일 하나로 폴더명만 모으는 최소 구현만 진행한다.** 경로 계산 함수의 추상화는 D(synthesize 단계 분리) 완료 이후 2차 작업으로 미룬다. 함수 추상화를 너무 이르게 진행하면 참조 관계가 얽혀 디버깅이 어려워진다.

대상 파일:

- 신규 상수 파일 1개 (`scripts/path_constants.py`)
- `scripts/generate.py`
- `scripts/synthesize.py`
- `scripts/review.py`
- `scripts/watcher.py`
- `scripts/scheduler.py`
- `scripts/dashboard/data_loader.py`
- `scripts/memory_manager.py`

수정 내용:

1. `path_constants.py`에 폴더명 상수를 정의한다.

```python
# path_constants.py
QUEUE_DIR      = "queue"
APPROVED_DIR   = "approved"
REJECTED_DIR   = "rejected"
QUIZ_SUBDIR    = "퀴즈"
SUMMARY_SUBDIR = "정리"
CACHE_DIR      = "cache"
OUTPUT_MD_DIR  = "output/md"
OUTPUT_PDF_DIR = "output/pdf"
```

2. 각 파일의 리터럴 문자열을 상수 참조로 교체한다.
3. 경로 계산 함수 자체는 이번에 이동하지 않는다.

완료 기준:

1. 위 폴더명 리터럴이 `path_constants.py` 하나에서만 정의된다.
2. 기존 동작이 바뀌지 않는다 (폴더명 변경이 아닌 참조 교체이므로).

### D. synthesize 처리 단계 분리

**배경 및 실제 문제:**

`synthesize.py`의 `process_note()`가 619줄 파일에서 200줄 이상을 차지하는 단일 함수다. 내부에서 `MemoryManager` 인스턴스가 두 번 별도 생성되고(line 425, 466), 일부 단계는 예외를 무시하고 일부는 중단하는 일관성 없는 오류 처리가 섞여 있다.

대상 파일:

- `scripts/synthesize.py`

수정 내용:

1. `process_note()` 내부를 단계 함수로 분리한다.

| 함수 | 역할 |
|------|------|
| `collect_sources_for_note` | 소스 텍스트 · 이미지 수집 |
| `run_agents` | 분류 · 갭 감지 · 교차과목 연결 |
| `run_paper_enrichment` | PubMed/arXiv 논문 보강 |
| `build_synthesis` | 핵심 합성 텍스트 생성 |
| `append_learning_sections` | 퀴즈 · 오답노트 섹션 추가 |
| `persist_outputs` | MD/PDF 저장 + vault 역주입 |

2. `MemoryManager` 인스턴스를 함수 최상단에서 한 번만 생성해 각 단계 함수에 전달한다.
3. 각 단계 함수의 예외 처리 방침을 통일한다: 단계 실패 시 해당 단계를 건너뛰고 로그에 단계명과 예외를 기록, `process_note()`가 실패 여부를 반환값으로 전달한다.

완료 기준:

1. `process_note()`가 오케스트레이션 역할만 수행한다.
2. 각 단계 함수가 명확한 입력과 반환값을 가진다.
3. `MemoryManager` 인스턴스가 `process_note()` 스코프에서 한 번만 생성된다.

### E. 로그 메시지 정리

**배경 및 실제 문제:**

`print()`와 `logger.xxx()`가 혼용되고, 실패 로그에 파일명과 단계명이 누락되어 어느 단계에서 실패했는지 파악하기 어렵다. D(synthesize 단계 분리) 전에 로그 패턴을 정해두면 분리된 함수 안에서 일관되게 적용할 수 있으므로 D보다 먼저 진행한다.

대상 파일:

- `scripts/watcher.py`
- `scripts/synthesize.py`
- `scripts/generate.py`
- `scripts/scheduler.py`

수정 내용:

1. 실행 시작 · 단계 시작 · 실패 · 저장 완료 로그 포맷을 통일한다.

```
[INFO]  {파일명} | {단계명} 시작
[OK]    {파일명} | {단계명} 완료
[WARN]  {파일명} | {단계명} 건너뜀 — {예외}
[ERROR] {파일명} | {단계명} 실패 — {예외}
```

2. `stdout`용 `print()` 문은 진행 표시용으로 유지하되, 오류·경고 메시지는 `logger`로 일원화한다.
3. 실패 시 파일명, 단계명, 예외 종류가 함께 보이게 맞춘다.

완료 기준:

1. 주요 단계 로그가 동일한 패턴을 가진다.
2. 실패 로그에서 최소한 파일명과 단계명을 확인할 수 있다.

```
A → B → C → E → D
```

| 순서 | 항목 | 이유 |
|------|------|------|
| 1 | A. weak_concepts 통일 | 저장 경로가 두 곳인 상태에서 다른 작업을 하면 데이터 일관성 보장 불가 |

| 4 | E. 로그 정리 | D 전에 로그 패턴을 고정해 단계 함수에서 즉시 적용 |
| 5 | D. synthesize 단계 분리 | 경로·로그 패턴이 고정된 뒤에 함수 추출 수행 |

## 기대 효과

1. `weak_concepts.json` 쓰기 경로가 `MemoryManager` 하나로 수렴한다.
2. CLI와 대시보드의 퀴즈 승인 결과가 동일해진다.
3. 대시보드에서 JSON 파싱 오류로 퀴즈가 무음 소실되지 않는다.
4. 폴더명 변경이 필요할 때 `path_constants.py` 한 파일만 수정하면 된다.
5. 2차 리팩터링(경로 함수 추상화, DB 도입) 전에 안정적인 기준선을 확보한다.

## 최종 판단

지금 단계에서 가장 현실적인 1차 수정은 아래 순서로 압축된다.

1. **상태 저장 통일** (A) — 데이터 일관성 리스크를 즉시 제거
2. **승인 처리 공용화** (B) — CLI·대시보드 행동 차이 + 무음 오류 제거
3. **경로 상수 통합** (C) — 최소 구현, 함수 추상화는 2차로 미룸
4. **로그 정리 → synthesize 분리** (E→D) — 패턴 고정 후 구조 개선

이 순서를 완료하면 이후의 큰 리팩터링(DB, 플러그인 구조)은 기능 추가가 아닌 구조 개선 작업으로 전환할 수 있다.
